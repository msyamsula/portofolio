# Backend App Context Summary

This document captures the architecture, structure, and key patterns used in the `backend-app` of the repository. It was generated by reading the `backend-app` folder and associated documentation. The goal is to keep this knowledge available for the AI assistant context.

## High-Level Overview

- The backend app is a Go monolith following Clean Architecture and Domain-Driven Design principles.
- It is organized into several top-level directories:
  - `binary/`: contains executables such as the HTTP server and other agents (e.g., pg-agent).
  - `domain/`: core business domains (user, friend, message, graph, url-shortener, agent, healthcheck).
  - `infrastructure/`: shared infrastructure code (database, HTTP setup, telemetry, etc.).
  - `mock/`: autogenerated or hand-written mocks for testing domain interfaces.
  - `docs/`: design docs, architecture guidelines, ADRs, and other documentation.

## Domain Structure and Patterns

Each domain (e.g., `user`, `friend`, `message`) follows a consistent internal layout:

```
<domain>/
  dto/           # request/response data contracts
  handler/       # HTTP handlers for the domain
  service/       # business logic and orchestration
  repository/    # data access abstractions (interfaces and implementations)
  integration/   # clients for external services (if any)
```

- **DTO**: defines structs used for input and output, keeping contracts explicit and decoupled.
- **Handler**: receives HTTP requests, performs validation, invokes services, and writes responses.
- **Service**: contains business logic and coordinates between repositories and external systems.
- **Repository**: declares interfaces for data access; implementations live in `infrastructure/database` or other infra packages.
- **Integration**: handles communication with external systems (e.g., gRPC, third-party APIs).

### Layer Responsibilities

| Layer         | Directory      | Responsibility                        |
|---------------|----------------|---------------------------------------|
| Presentation  | `handler/`     | HTTP request handling and validation  |
| Application   | `service/`     | Business rules and orchestration      |
| Persistence   | `repository/`  | Data access abstraction               |
| Contracts     | `dto/`         | Request/response definitions          |
| Integration   | `integration/` | External service clients              |

## Infrastructure

- `infrastructure/database/`: contains DB setup, connection pooling, migration helpers.
- `infrastructure/http/`: bootstraps HTTP server, routers, middleware.
- `infrastructure/telemetry/`: telemetry client and instrumentation utilities (OpenTelemetry, metrics, logs).

## Executables

- The main HTTP server lives under `binary/http` with its own `main.go` and Dockerfile.
- Additional binaries (e.g., `pg-agent`) exist for background tasks.

## Documentation

The `docs/` directory holds numerous markdown files describing architecture decisions, clean architecture, repository pattern, domain-driven design, etc. These provide guidance for contributing and maintaining the codebase.

## Testing and Mocks

- Mocks for domain interfaces are present in `mock/`; often generated using tools like mockery or manually defined.
- Each domain directory includes `_test.go` files for unit tests on handlers, services, and repositories.

## Observability and Deployment

- Observability is a first-class concern with telemetry instrumentation across domains.
- Deployment artifacts (Dockerfiles, Kubernetes manifests) are scattered in relevant directories (e.g., `binary/http` for Dockerfile).
